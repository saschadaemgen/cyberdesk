## What CyberDesk is

A single fullscreen application in the style of a serious cyber operating system: fixed zone layout, one color world (base #04070A, brand blue #009FE3), heavily animated, optimized for ultrawide displays (target roughly 1.20 m of screen width, 16:9 as fallback). Only CARVILON-built applications run inside it, plus one surf zone. No freely movable windows - predictability is the core of the product.

## Layer model

1. **Zones (fixed):** Spine on the left, main area (three sizes S/M/L with reflow of neighboring zones), video zone, terminal zone, right tab rail (Status, Files, FTP, Music, ...). Positions are law. Changes only in Edit Mode within the grid rules (D-0007).
2. **Modes (layout presets):** Standard, Admin (larger main area), Do-Not-Disturb, and others. A mode loads a preset of the same fixed layout - it never invents a new one.
3. **Event Priority Engine:** Events (doorbell, call, alarm, security alert) carry priorities, zones carry ranks, the mode acts as the gate. Decision per event: override, overlay, or suppress. Even interruptions are regulated.

## Process and technology model

- **Rust host:** window management (winit), rendering (wgpu), zones/modes/event engine - and, since CD-40, the vault crypto core (`src/vault.rs`, D-0058; authoritative unlock model CD-42, D-0062): envelope key management around one random Vault Master Key. The **master password is the mandatory sole root** (Argon2id envelope; RFC 9106 params, re-tunable); a WebAuthn-PRF passkey is the only optional additional factor; **no recovery key exists**. The unlock policy is structural and exactly two-shaped - password-only (`{password}` envelope) or password + passkey 2FA (`{password, passkey}` pair envelope); every envelope contains the password, so a passkey can never replace it (XChaCha20-Poly1305 wrapping; sealed app state under the VMK; `VirtualLock`ed + zeroized key memory). The start-authorization boot gate is LIVE (Stage 1b, D-0059; CD-42 Task A): the shell boots into `cyberdesk://lock/` only - unlocking an existing vault, or the MANDATORY first-launch master-password setup when none exists - no slots, MF zone, or HUD exist until the gate opens, with the HOST capturing the keystrokes into locked memory (the page renders a masked count; no renderer ever holds a secret keystroke). The setup/change flows carry a HOST-computed live strength meter (vetted `zxcvbn` crate, 64-char eval cap): only score/criteria/canned-feedback cross to the page, and a weak submit parks on a prominent warning until the explicit "use it anyway" IPC - the informed override, host-revalidated. The passkey is LIVE via **Windows Hello / WebAuthn PRF** (`src/webauthn.rs`, CD-43, D-0063): the Win32 `webauthn.dll` API through `windows-sys 0.61.2` as a direct dependency (raw-dylib, no import lib; the v7 header's API-v4-era hmac-secret salt path - no v8 FFI needed, per the Task-0 determination). Enrollment runs from the unlocked session on a vault worker (modal MakeCredential with the hmac-secret extension + `bEnablePrf`, then the first PRF eval assertion - two Hello prompts by CTAP design); the OS applies the WebAuthn-spec salt hashing to a per-passkey random eval value persisted non-secret on the method (alongside the credential id). The 2FA unlock flow gathers the host-captured password, then runs the Hello assertion (`hello: "assert"` rides the state push for the page hint), and both factors together open the `{password, passkey}` pair envelope; a failed/cancelled Hello step preserves the typed password (locked memory) for a one-Enter retry. Output structs are read only up to the RETURNED `dwVersion` (the kanidm #262 Hello-crash lesson); the PRF secret lives in `SecretBuf` pages and the DLL's output buffer is zeroized before `WebAuthNFreeAssertion`. Removing the passkey re-mints the `{password}` envelope and best-effort-deletes the OS credential (`WebAuthNDeletePlatformCredential`) - never a brick. Unlock/setup derivations run on worker threads; the workspace boot (identity seed from the sealed store, then the normal session restore) follows the unlock/setup outcome; "Lock now" wipes key material and relaunches the process cold; the dev bypass exists only under `debug_assertions`. A v1 (recovery-key model) `vault.json` is refused with a reset message - no migration, per the CD-42 briefing (dev data only).
- **Settings is a full-screen layer, not a card (CD-44 Stage C, D-0064):** the gear gives the internal view the whole window, and the workspace is DISPLACED rather than covered - slots, both side zones and the HUD strip travel downward by an eased, smoothstep-shaped offset (`settings_slide`, eased both ways per frame), applied to the COMPOSITED rects only so the animated frame and input routing keep their real geometry. The Pulse Grid background is drawn independently and never moves, so the layer reads as the settings pushing the workspace aside. The page itself is a card grid packed into balanced multi-columns (four at 1920, more on an ultrawide), on one rem scale (`clamp(14px, 0.78vw, 19px)`) that holds from 1920x1080 out to roughly 5200 wide, with each section's long explanation behind a collapsed disclosure so nothing needs a scrollbar. Per-section reveal animation is a later addition: each card is already an independent block.
- **The gate's input model (CD-40 1b, ergonomics corrected in CD-44 A1):** while the lock/setup view is up the HOST owns the keyboard, so the page renders a masked count and NEVER an input element. Within that constraint the field behaves like a real password field: typing, Backspace, Ctrl+V paste (a host clipboard read), Enter to submit, a visible live-focus ring and a neutral placeholder. Escape is non-destructive: it clears the entry, or steps back exactly one step when the entry is empty, and the mandatory flows (first-launch setup, unlock) never abort. Text selection and caret placement genuinely cannot exist under host capture; the copy says the field is already focused instead of faking them.
- **CEF (Chromium Embedded Framework):** delivers pixels of the surf zone, nothing else. CD-01: windowed embed as feasibility proof; from CD-02 on, offscreen rendering into a GPU texture, then feathering/compositing inside our own frame (soft edges bleeding into the design). CEF runs with an isolated browser profile (own root_cache_path) - the surf zone never shares state with any user-installed browser.
- **Hard process boundary host<->CEF**, IPC only through an explicit allowlist. No Electron, no Node, no npm chain in the core. The Chromium **OS sandbox is currently deferred** (D-0008: the cef-rs Windows sandbox requires the bootstrap.exe launcher model, which breaks the plain cargo-run path - a time-boxed deviation with a hard re-enablement gate before Season 5/6); the doctrine remains "sandbox active", and the boundary today is the process split + the scheme-locked IPC allowlist.
- **NetGuard:** no module opens connections on its own; everything goes through the central network layer (deny-by-default per zone, certificate pinning, own DNS resolver, kill switch, counters). Browser traffic attaches to the same monitor via CefRequestHandler. **One documented exception exists:** (D-0027, CD-15; extended to onion services by D-0052, CD-35) per-window Tor traffic - user-driven browsing through the Tor network when a window is toggled to Tor (embedded arti, a local per-slot SOCKS relay). The earlier second exception - the CD-13 pinned update-manifest fetch (D-0023) - was **retired in CD-22 (D-0036)**: the update surface is client-side now and the host opens no HTTP client of its own. Tor is the host's only intentional outbound path until NetGuard is built, when it becomes its first allowlist entry.

## Platform path

Development: Windows 11 (MSVC). Later: Linux appliance. Long-term goal: CARVILON OS (Debian 13 "Trixie") booting directly into CyberDesk as its shell - the app is the first deliverable and the later heart of the OS. Nothing on the app path is throwaway work.
